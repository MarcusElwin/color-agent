# Releasing

The release pipeline (`.github/workflows/release.yml`) is fully automated once the manual one-time setup is done.

## One-time setup

### 1. Reserve the project on PyPI and TestPyPI

Trusted Publishing requires the project to exist before the first publish. Easiest path:

```bash
# Build locally
python -m build
# Upload once with an API token (TestPyPI first to validate)
twine upload --repository testpypi dist/*
twine upload dist/*
```

Or, if you don't want to do a manual upload, create the project shell on each registry via the web UI.

### 2. Configure Trusted Publishers (no API token needed after this)

For each of **PyPI** (https://pypi.org/manage/account/publishing/) and **TestPyPI** (https://test.pypi.org/manage/account/publishing/):

| Field | Value |
|---|---|
| PyPI Project Name | `color-agent` |
| Owner | `MarcusElwin` |
| Repository name | `color-agent` |
| Workflow name | `release.yml` |
| Environment name | `pypi` (or `testpypi` for the test registry) |

### 3. Create the GitHub environments

In the repo: **Settings → Environments → New environment**.

Create two: `pypi` and `testpypi`. Optionally add protection rules (require manual approval to deploy, restrict to `main` and tags) on the `pypi` environment for safety.

## Cutting a release

1. Update `pyproject.toml` `version` and `CHANGELOG.md`.
2. Commit, PR, merge to `main`.
3. Tag and push:

   ```bash
   git tag v0.2.0
   git push origin v0.2.0
   ```

4. The `release` workflow runs:
   - Tests on Python 3.12 (gate).
   - Verifies the tag matches `pyproject.toml` (gate).
   - Builds sdist + wheel, runs `twine check`.
   - Publishes to PyPI via OIDC.
   - Creates a GitHub Release with autogenerated notes and the dist files attached.

## Dry-run a release without tagging

Push a commit, then **Actions → release → Run workflow → target: testpypi**. This builds and publishes to TestPyPI only, so you can install and validate before cutting a real tag:

```bash
pip install -i https://test.pypi.org/simple/ color-agent
```
